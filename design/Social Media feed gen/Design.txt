Part 2: Push vs Pull Models — The Fundamental Choice
Aadvik: “When a user posts, how does it get into their followers’ feeds? Push or pull?”

Sara: “This is the core architectural decision. Let me explain both approaches:”

Option 1: Pull Model (Fan-out on Read)
How it works:

When user requests feed, fetch posts from all users they follow
Query database: “Get recent posts from users I follow”
Aggregate and rank results
Return feed
Flow:

User A requests feed
→ Query: SELECT * FROM posts WHERE user_id IN (followed_users) ORDER BY created_at DESC LIMIT 50
→ Aggregate results
→ Rank by engagement/relevance
→ Return feed
Pros:

Simple to implement
No pre-computation needed
Easy to change ranking algorithm (just change query)
Low write overhead (only store post once)
Good for users with few followers
Cons:

Expensive for users with many follows (need to query many users’ posts)
Slower reads (need to aggregate on-the-fly)
Hard to cache (each user’s feed query is unique)
Doesn’t scale well for users following 1000+ accounts”
Aadvik: “What about push?”

Option 2: Push Model (Fan-out on Write)
How it works:

When user posts, immediately push post to all followers’ feed caches
Pre-compute and store feed entries for each follower
On feed request, just read pre-computed feed
Flow:

User B posts
→ For each follower of User B:
    → Add post to follower's feed cache
→ Done
User A requests feed
→ Read from pre-computed feed cache
→ Return feed (already ranked, ready to go)
Pros:

Fast reads (feed is pre-computed, just read from cache)
Easy to cache (feed already computed)
Great for users with many followers (fan-out happens once on write)
Real-time updates (feed updated immediately)
Cons:

Expensive writes (100M writes for celebrity post)
Wasted work for inactive users (may never read their feed)
Hard to change ranking algorithm (need to recompute all feeds)
Complex invalidation (what if user unfollows?)”
Aadvik: “So which do you choose?”

Sara: “I’d recommend a Hybrid Approach — use push for active users with reasonable follower counts, and pull for power users with millions of followers.

Hybrid Strategy:

Push Model — For users with < 1M followers (99.9% of users)
Pull Model — For celebrities with > 1M followers (0.1% of users)
Why Hybrid?

Most posts are from regular users → Push works great (fast reads)
Celebrity posts are rare but expensive → Pull on-demand (avoid 100M writes)
Best of both worlds”

Part 4: Caching Strategies
Aadvik: “Caching is critical for read-heavy workloads. How do you cache feeds?”

Sara: “We need multi-layer caching:

Layer 1: Pre-computed Feed Cache (Redis)

Store user’s feed as sorted set
Updated in real-time as posts arrive
TTL: 24 hours (refresh daily)
Layer 2: API Response Cache (Redis/In-memory)

Cache rendered feed response (JSON)
TTL: 1–5 minutes (balance freshness vs load)
Invalidate on new posts
Layer 3: Post Content Cache

Cache post details (author, content, engagement)
Shared across all users (one post cached, used in many feeds)
TTL: 15 minutes”

Sara: “Cache invalidation strategy:

For Feed Cache (Layer 2):

New post arrives → Fan-out adds to followers’ feed caches
Feed cache automatically updated (no invalidation needed)
Response cache (Layer 1) invalidated for affected users
For Post Content Cache (Layer 3):

Post updated → Invalidate post cache
Engagement changes → Update cache (likes, comments)

Pre-warm active users — Pre-load feeds for top 10% active users
Predictive warming — Warm feeds before peak hours
Lazy loading — Load first 50 posts on demand, prefetch next 50
Background refresh — Refresh feeds in background during low traffic”
Part 5: Content Ranking and Relevance
Aadvik: “Users don’t just want chronological feeds. They want relevant content. How do you rank posts?”

Sara: “Content ranking is complex. We need to balance multiple factors:

Ranking Factors:

Recency — Newer posts rank higher
Engagement — Likes, comments, retweets boost ranking
Author relevance — Posts from users you interact with more
Content type — Videos might rank higher than text
User preferences — Learned from past engagement
Ranking Score Formula:

Score = (recency_score × 0.3) + 
        (engagement_score × 0.4) + 
        (author_relevance × 0.2) + 
        (content_boost × 0.1)
Aadvik: “How do you compute this efficiently for millions of posts?”

Sara: “We pre-compute scores during fan-out:

On post creation — Calculate initial score (based on recency, author)
During fan-out — Add to feed cache with score
On engagement — Recalculate score, update in cache
Periodic refresh — Recalculate scores for older posts”

Part 7: Database Design
Aadvik: “How do you store posts and relationships in the database?”

Sara: “We need multiple tables for different purposes:”

Database Schema
Posts Table:

CREATE TABLE posts (
    post_id BIGINT PRIMARY KEY,
    user_id VARCHAR(50) NOT NULL,
    content TEXT,
    post_type ENUM('TEXT', 'IMAGE', 'VIDEO', 'RETWEET') NOT NULL,
    parent_post_id BIGINT NULL,  -- For retweets/comments
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP,
    
    -- Engagement metrics (denormalized for fast reads)
    like_count INT DEFAULT 0,
    comment_count INT DEFAULT 0,
    retweet_count INT DEFAULT 0,
    view_count BIGINT DEFAULT 0,
    
    -- Indexes
    INDEX idx_user (user_id, created_at DESC),
    INDEX idx_created (created_at DESC),
    INDEX idx_engagement (like_count DESC, created_at DESC)
);
Follows Table:

CREATE TABLE follows (
    follower_id VARCHAR(50) NOT NULL,
    followee_id VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL,
    PRIMARY KEY (follower_id, followee_id),
    INDEX idx_follower (follower_id),
    INDEX idx_followee (followee_id)
);
Engagements Table (Likes, Comments):

CREATE TABLE engagements (
    engagement_id BIGINT PRIMARY KEY,
    post_id BIGINT NOT NULL,
    user_id VARCHAR(50) NOT NULL,
    engagement_type ENUM('LIKE', 'COMMENT', 'RETWEET') NOT NULL,
    content TEXT NULL,  -- For comments
    created_at TIMESTAMP NOT NULL,
    UNIQUE KEY (post_id, user_id, engagement_type),
    INDEX idx_post (post_id, engagement_type),
    INDEX idx_user (user_id, created_at DESC)
);
Aadvik: “Why denormalize engagement counts in the posts table?”

Sara: “Performance optimization:

Without denormalization: Every feed read needs COUNT(*) queries for likes/comments → Expensive
With denormalization: Counts stored directly → Fast reads
Trade-off: Need to update counts on every engagement (acceptable write overhead for massive read benefit)
We update counts asynchronously to avoid blocking engagement actions.”

Part 10: Database Choice Deep Dive — SQL vs NoSQL
Aadvik: “Let’s have a deeper discussion about database choices. When would you use SQL vs NoSQL?”

Sara: “This is a critical decision. Let me break down the use cases:”

Use Case Analysis
Posts Storage:

Access Pattern: Write-heavy (1B posts/day), read-heavy (100B reads/day)
Query Pattern: Range queries by timestamp, user_id, engagement metrics
Consistency: Eventual consistency acceptable for feed, strong consistency for post details
Follows/Relationships:

Access Pattern: Read-heavy (get followers list frequently)
Query Pattern: Simple lookups, graph queries (mutual follows)
Consistency: Strong consistency needed (can’t show wrong follower count)
Engagements:

Access Pattern: Write-heavy (billions of likes/comments/day)
Query Pattern: Aggregations (count by post_id), lookups
Consistency: Eventual consistency acceptable (counts can lag slightly)”
Option 1: Pure SQL (PostgreSQL)
Posts Table (Detailed):

CREATE TABLE posts (
    post_id BIGSERIAL PRIMARY KEY,
    user_id VARCHAR(50) NOT NULL,
    content TEXT NOT NULL,
    post_type VARCHAR(20) NOT NULL CHECK (post_type IN ('TEXT', 'IMAGE', 'VIDEO', 'RETWEET')),
    parent_post_id BIGINT NULL,
    
    -- Timestamps
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    
    -- Denormalized engagement counts (updated async)
    like_count INT DEFAULT 0 NOT NULL,
    comment_count INT DEFAULT 0 NOT NULL,
    retweet_count INT DEFAULT 0 NOT NULL,
    view_count BIGINT DEFAULT 0 NOT NULL,
    
    -- Ranking score (denormalized, updated by ranking service)
    ranking_score DECIMAL(5,4) DEFAULT 0.5000,
    
    -- Indexes
    CONSTRAINT fk_parent_post FOREIGN KEY (parent_post_id) REFERENCES posts(post_id),
    INDEX idx_user_created (user_id, created_at DESC),
    INDEX idx_created_desc (created_at DESC),
    INDEX idx_ranking (ranking_score DESC, created_at DESC),
    INDEX idx_engagement (like_count DESC, created_at DESC),
    
    -- Full-text search index
    INDEX idx_content_fulltext USING gin(to_tsvector('english', content))
);

Part 11: Ranking Service Deep Dive
Aadvik: “Let’s dive deep into the ranking service. How exactly does ranking work?”

Sara: “Ranking is the heart of personalized feeds. Let me explain the algorithms:”

Ranking Algorithm Components
1. Recency Score:

recency_score = exponential_decay(timestamp_now - post_timestamp)
Formula: e^(-λ * hours_old)
Where λ (decay factor) = 0.1 (configurable)

Examples:
- 1 hour old: e^(-0.1 * 1) = 0.90
- 6 hours old: e^(-0.1 * 6) = 0.55
- 24 hours old: e^(-0.1 * 24) = 0.09
- 48 hours old: e^(-0.1 * 48) = 0.008
2. Engagement Score:

engagement_score = normalize(
    (like_count × like_weight) + 
    (comment_count × comment_weight) + 
    (retweet_count × retweet_weight) + 
    (view_count × view_weight)
)

Weights (configurable):
- like_weight: 1.0
- comment_weight: 3.0 (comments more valuable)
- retweet_weight: 5.0 (retweets most valuable - viral signal)
- view_weight: 0.1 (views less valuable, easy to game)
Normalization: Use log scale to prevent viral posts from dominating
normalize(x) = log(1 + x) / log(1 + max_engagement)
3. Author Relevance Score:

author_relevance = calculate_relationship_strength(author_id, viewer_id)

Factors:
- Interaction frequency (how often viewer engages with author)
- Mutual connections (common follows)
- Follow duration (longer = more relevant)
- Engagement rate (viewer's past engagement with author's posts)


Formula:
author_relevance = (
    interaction_score × 0.4 +
    mutual_connections_score × 0.2 +
    follow_duration_score × 0.2 +
    past_engagement_rate × 0.2
)
4. Content Boost Score:

content_boost = calculate_content_quality(post)

Factors:
- Post type (videos might get 1.2x boost, images 1.1x, text 1.0x)
- Media quality (high-res images, HD videos)
- Hashtags (trending hashtags boost)
- Content length (optimal length range gets boost)
- Spam detection (spam posts get 0.0 boost)
Complete Ranking Formula
final_score = (
    recency_score × 0.30 +
    engagement_score × 0.40 +
    author_relevance × 0.20 +
    content_boost × 0.10
)

Final score range: 0.0 to 1.0

CREATE MATERIALIZED VIEW user_author_relevance AS
SELECT 
    viewer_id,
    author_id,
    COUNT(DISTINCT post_id) as interaction_count,
    SUM(CASE WHEN interaction_type = 'LIKE' THEN 1 ELSE 0 END) as like_count,
    SUM(CASE WHEN interaction_type = 'COMMENT' THEN 1 ELSE 0 END) as comment_count,
    SUM(CASE WHEN interaction_type = 'RETWEET' THEN 1 ELSE 0 END) as retweet_count,
    MAX(created_at) as last_interaction_at,
    MIN(created_at) as first_interaction_at
FROM user_interactions
GROUP BY viewer_id, author_id;

Part 14: Trade-offs and Final Thoughts
Aadvik: “Let’s summarize the key trade-offs we made.”

Sara: “Here are the major trade-offs:

Critical Trade-offs
1. Push vs Pull Model

Chose: Hybrid (Push for regular, Pull for celebrities)
Trade-off: Push gives fast reads but expensive writes. Pull gives cheap writes but slow reads. Hybrid optimizes for common case.
2. Ranking: Pre-compute vs On-demand

Chose: Pre-compute during fan-out
Trade-off: Pre-computation increases write cost but makes reads instant. On-demand ranking slows reads but reduces storage.
3. Cache Invalidation: Immediate vs TTL-based

Chose: TTL-based with updates
Trade-off: Immediate invalidation ensures freshness but complex. TTL-based simpler but eventual consistency.
4. Feed Storage: Database vs Cache

Chose: Redis for active feeds, Database for persistence
Trade-off: Redis fast but expensive. Database cheaper but slower. Hybrid gives speed where needed.
5. Fan-out: Synchronous vs Asynchronous

Chose: Asynchronous with Kafka
Trade-off: Synchronous ensures immediate updates but blocks. Asynchronous non-blocking but slight delay acceptable.
Key Design Principles
Write-Heavy Optimization — Design for efficient fan-out (majority of load)
Cache-First — Aggressive caching for read-heavy workload
Async Processing — Use queues for non-critical path
Horizontal Scalability — Every component scales independently
Graceful Degradation — System works even if cache fails (fallback to DB)
